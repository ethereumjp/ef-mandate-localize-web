# コントリビューションガイド (Draft)
*English follows.

## 目的

このプロジェクトは、Ethereum Foundation Mandate の「忠実な日本語公式翻訳」を作ることを目的としません。Mandate の理解を助け、議論のきっかけを作ることを目的としています。

ゴールは、日本語話者のコントリビューターが Mandate を読み、必要に応じて翻訳を修正し、文脈を補足し、問いを立て、日本の Ethereum コミュニティの視点から解釈や論点を可視化することです。

また、それらを踏まえて、EF とコミュニティ、またコミュニティ間でどのように対話できるかを可視化することを目指します。

そのため、Mandate の translation / localization は一度完了して終わるものではありません。時代や文脈の変化に応じて、常に更新され続けるものとして扱います。

## 基本原則

### 1. 忠実翻訳を目指さない

単なる直訳ではなく、日本語として読める形にしつつ、必要に応じて解釈・問い・補足を加えることを許容します。

### 2. EF の公式翻訳として扱わない

このプロジェクトは、EF の承認文書を作る場ではありません。

EF の意図を忖度して訳すのではなく、参加者がどのように読んだかをベースにします。

### 3. 原文への敬意を保つ

原文の論点をねじ曲げないようにします。

ただし、読者の理解を助けるためのコメント、疑問、批判的視点の追加は許容します。

### 4. ローカル文脈を含めてよい

日本コミュニティから見た含意、違和感、期待、論点を明示してよいものとします。

## コントリビューションの種類

各 contribution は、少なくとも次のいずれかに分類します。

### Question

「なぜこの表現なのか」「この範囲は何を指すのか」などの問い。

### Commentary

読み手としての解釈、補足、論点提示。

### Critique

違和感、異論、懸念の明示。

### Localization note

日本コミュニティの文脈での意味づけや読み替え。

### Clarification

主に書き手による、用語や文脈の説明。

## 書き方ルール

* 1 contribution = 1 論点とします。
* できるだけ、対応する原文の一節を明示してください。
* 事実説明と意見は分けてください。
* 解釈を加える場合は、**Question**、**Commentary**、**Critique**、**Localization note**、**Clarification** のいずれかだと分かるラベルを付けてください。
* 「EF はこう言いたいはず」のような断定は避けてください。
* 「私は / 私たちはこう読んだ」のように、主語を明確にしてください。

## GitHub 上の運用ルール案

* PR / Issue の使い分けを明確にします。

  * **PR**: 文面変更提案
  * **Issue**: 質問・論点の提起

* 各投稿には以下を含めてください。

  * 対象箇所
  * contribution type
  * 内容
  * 必要であれば理由

## 合意形成

* 全員一致を必須にしません。
* 意見が割れる場合は、無理に統合せず、並記を許容します。
* 「コミュニティとして単一見解を出す」ことよりも、複数の読みを残すことを優先します。

## 併記の扱い

意見が割れる場合は、無理に単一の文面に統合せず、Markdown 本文または Issue 上で複数案を併記します。

本文に残す場合は、「読み A / 読み B」「Interpretation A / Interpretation B」などの見出しを付け、どの立場からの解釈かを明示してください。

補足的または長い意見は、必要に応じて <details> 解釈1:...<br>解釈2:... </details> を使って折りたたんでください。

## トーンと行動規範

議論では、以下を心がけてください。

* 敬意を持つ
* 具体的に書く
* 建設的に提案する
* 異なる解釈に開かれている

以下は避けてください。

* 個人攻撃、侮辱、嘲笑、嫌がらせ
* 参加者全体を代表するかのような断定
* 特定の参加者やグループに対する決めつけ
* 議論と無関係な広告、宣伝、営業、勧誘
* 商品、サービス、イベント、コミュニティ、トークン等への不適切な誘導
* スパム投稿、同一内容の繰り返し投稿
* 論点から意図的に話を逸らす行為
* 相手の主張を意図的に歪めて批判すること
* 根拠のない断定や、確認できない情報を事実として扱うこと
* 非公開情報、個人情報、機密情報の共有
* 他者になりすます行為
* 脅迫、威圧、過度に攻撃的な表現
* 議論の勝敗を目的化すること
* 合意がないにもかかわらず、単一の見解としてまとめること
* 複数の解釈や少数意見を不当に排除すること

# Contribution Guide (Draft)

## Purpose

This project does not aim to create an “official Japanese translation” of the Ethereum Foundation Mandate. Its purpose is to support understanding of the Mandate and create opportunities for discussion.

The goal is to enable Japanese-speaking contributors to read the Mandate, revise the translation where necessary, add context, raise questions, and make interpretations and discussion points visible from the perspective of the Japanese Ethereum community.

It also aims to make visible how the EF and the community, as well as different parts of the community, can engage in dialogue based on those interpretations and discussion points.

For this reason, the translation / localization of the Mandate is not something that will be completed once and then considered finished. It should be treated as something that is continuously updated over time, as contexts and circumstances change.

## Basic Principles

### 1. Do not aim for a faithful translation

This project does not aim for a simple literal translation. Contributions may adapt the text so that it reads naturally in Japanese, and may add interpretations, questions, or supplementary context where appropriate.

### 2. Do not treat this as an official EF translation

This project is not a place to produce an EF-approved document.

Contributors should not translate based on assumptions about what the EF “must have intended.” Instead, contributions should be based on how participants read and understand the text.

### 3. Maintain respect for the original text

Contributors should not distort the points made in the original text.

However, comments, questions, and critical perspectives may be added where they help readers better understand the text.

### 4. Local context may be included

Contributors may explicitly note implications, discomfort, expectations, and discussion points from the perspective of the Japanese community.

## Types of Contributions

Each contribution should be classified as at least one of the following.

### Question

Questions such as “Why is this expression used?” or “What does this scope refer to?”

### Commentary

Interpretations, supplementary explanations, or discussion points from the reader’s perspective.

### Critique

Explicit statements of discomfort, disagreement, or concern.

### Localization note

Contextualization or reframing based on the perspective of the Japanese community.

### Clarification

Explanations of terms or context, mainly provided by the author.

## Writing Rules

* One contribution should address one point.
* Where possible, clearly identify the corresponding passage in the original text.
* Separate factual explanations from opinions.
* When adding interpretation, label it clearly as **Question**, **Commentary**, **Critique**, **Localization note**, or **Clarification**.
* Avoid definitive statements such as “The EF must be trying to say this.”
* Make the subject clear, such as “I read this as...” or “We read this as...”

## Proposed GitHub Workflow

* Clearly distinguish between PRs and Issues.

  * **PR**: Proposed text changes
  * **Issue**: Questions or discussion points

* Each post should include the following:

  * Target passage
  * Contribution type
  * Content
  * Reason, if necessary

## Consensus Building

* Unanimous agreement is not required.
* When opinions differ, it is acceptable to keep multiple views side by side rather than forcing them into a single unified position.
* The priority is not to produce a single “community position,” but to preserve multiple readings.

## Handling Multiple Views

When opinions differ, contributors should not force them into a single unified wording. Multiple views may be kept side by side in the Markdown text or in an Issue.

When multiple views are included in the main text, use headings such as “Reading A / Reading B” or “Interpretation A / Interpretation B,” and make clear which perspective each interpretation comes from.

Supplementary or longer opinions may be placed inside <details> Interpretation A:... <br> Interpretation B:... </details> blocks where appropriate.

## Tone and Code of Conduct

In discussions, please aim to be:

* Respectful
* Specific
* Constructive
* Open to different interpretations

Please avoid:

* Personal attacks, insults, mockery, or harassment
* Statements that appear to represent all participants as a whole
* Assumptions or stereotypes about specific participants or groups
* Advertisements, promotions, sales activity, or solicitation unrelated to the discussion
* Inappropriate promotion of products, services, events, communities, tokens, or similar items
* Spam or repeated posting of the same content
* Intentionally derailing the discussion
* Misrepresenting someone else’s argument in order to criticize it
* Presenting unsupported claims or unverified information as fact
* Sharing non-public information, personal information, or confidential information
* Impersonating others
* Threats, intimidation, or overly aggressive language
* Treating the discussion as something to be won or lost
* Presenting a single view as the shared position when no such agreement exists
* Unfairly excluding multiple interpretations or minority views
